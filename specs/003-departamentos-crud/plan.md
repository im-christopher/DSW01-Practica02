## Technical Context

**Language/Version**: Java 17 (mandatory - per constitution)  
**Primary Dependencies**: Spring Boot 3.x, Spring Security, Spring Data JPA, springdoc-openapi  
**Storage**: PostgreSQL (mandatory - shared database with empleados)  
**Testing**: JUnit 5 + Spring Boot Test (unit + integration)  
**Target Platform**: Linux container runtime (Docker - same containers as empleados app)  
**Project Type**: Backend web-service (REST API only, no frontend)  

**Performance Goals**: 
- Creación de departamento < 5 segundos (incluyendo validación de unicidad)
- Soportar mínimo 1000 departamentos sin degradación
- Consultas de empleados con departamento < 500ms para datasets de hasta 10,000 empleados
- 100% de eliminaciones con empleados asignados rechazadas correctamente
- 0% tasa de error con datos válidos

**Constraints**: 
- Basic Auth requerido en todos los endpoints `/api/v1/departamentos/**`
- Swagger contract requerido para documentación de nuevos endpoints
- Secrets via environment variables (no hard-coding)
- Soft delete obligatorio (campo `activo`, no DELETE físico)
- FK nullable en empleados (permitir empleados sin departamento)
- Nombre de departamento limitado a 100 caracteres
- ON DELETE RESTRICT en FK (prevenir eliminación de departamento con empleados)
- Usar mismo pool de conexiones PostgreSQL existente

**API Versioning**: Endpoints bajo `/api/v1/departamentos` (mantiene versioning existente, sin cambios breaking)  

**Scale/Scope**: 
- 3 User Stories: US1 (CRUD MVP), US2 (Assign employees), US3 (Query by department)
- 37 tareas de implementación en 6 fases
- MVP alcanzado con Fases 1+2+3 (22 tareas)
- Entidades: Nueva tabla `departamentos`, extensión de tabla `empleados`
- Endpoints: 5 nuevos (POST, GET, GET /{id}, PUT /{id}, DELETE /{id}) + 1 extensión (GET /empleados con filtro)

## Summary
│   ├── pages/
Esta feature implementa gestión completa de departamentos (CRUD) con soft delete y establece relación many-to-one entre empleados y departamentos. Los departamentos se ordenan alfabéticamente por nombre, no requieren control de versión optimista, y usan el mismo patrón de paginación que empleados. Las respuestas de empleados incluyen información del departamento asignado (solo id y nombre). La eliminación está bloqueada si el departamento tiene empleados asignados.

**Technical Approach**: Flyway migration (V3) para crear tabla `departamentos` y agregar FK `departamento_id` a `empleados`. Implementado como Spring Boot REST API bajo `/api/v1/departamentos` con autenticación Basic Auth. Usa soft delete (campo `activo`) en lugar de eliminación física.
